# DD-008 — Task aggregation, dedup, cap, and archive

Status: **accepted** (implementation immediately follows)
Predecessors: DD-002 (3-layer pipeline), DD-003 (artifacts/blockers)
Trigger: 2026-05-15 — user observed claude-stray's tasks list
missed recent work (DD-005/006/007, ARCHITECTURE rewrite, commits)
while `progress` was accurate. Investigation showed Layer 1 summaries
contain the tasks but Layer 2 has no rule to mine, dedup, cap, or
archive them.

## 1 — Problem

Comparing `dashboard.json:tasks[]` against `cache/summaries/<sid>.md` for
the claude-stray initiative on 2026-05-15:

- Summary's `# 任务（建议）` section had 7 clean checkbox tasks for
  today's design-doc work.
- `dashboard.json.tasks[]` had 43 entries, none of them from today.
- 4+ pairs of obvious near-duplicates ("P14.2: 写 prompts/... + 调优"
  appearing twice with subtly different wording).

Two failures rolled into one symptom:

**Failure A — "Missing"**: `prompts/classify-cross-session.md` only
says (workflow step 2, hot case) "you may update `progress`, refresh
tasks (done monotone)". No rule tells AI to read each summary's
`# 任务` section and merge new entries. Compared to DD-003's §7a /
§7b for artifacts / blockers (detailed: union, dedup by url,
most-recent-wins, sort, cap), tasks are an architectural
afterthought.

**Failure B — "Duplicated"**: tasks have no stable identifier — only
free-text `title`. When the title is reworded across rounds, AI
correctly preserves the old (continuity: no-deletion) AND adds the
"new" one. Over weeks this snowballs.

**Why `progress` doesn't suffer**: `progress` is regenerated fresh
every round with no continuity constraint. AI rewrites it freely.

## 2 — Goals

1. **Mine tasks systematically**: every hot summary's tasks reach
   their initiative's `tasks[]`.
2. **Stable identity** for each task so dedup works deterministically.
3. **Done-monotone** preserved (continuity invariant).
4. **Cap visible list** so cards don't accumulate dozens of stale
   completed entries; default `MAX_VISIBLE_TASKS = 20`.
5. **Archive everything** to disk so historical context isn't lost.
   Users can view the full history per initiative.
6. **Deterministic post-process in `classify.py`** so AI drift doesn't
   break invariants (mirrors what we already do for cold initiatives
   and for artifacts).

Non-goals:
- Cross-initiative task identity (each initiative owns its task ids)
- Task-level natural-language fuzzy-merge (v1 keeps exact-slug dedup;
  v2 could add normalized-form fuzzy match — see §6.2)
- User editing individual archived tasks
- AI ranking of task importance

## 3 — Design

### 3.1 — Task identity: stable slug

Each task gets a stable `id` derived deterministically from its
`title`:

```python
def slugify_task_title(title: str, max_len: int = 64) -> str:
    """Slug: lowercase, replace non-word chars with '-', collapse
    repeats, trim, cap length. Stable across runs for the same title."""
    import re
    s = re.sub(r'\s+', ' ', title.strip().lower())
    s = re.sub(r'[^\w]+', '-', s, flags=re.UNICODE)   # keep CJK as \w
    s = re.sub(r'-+', '-', s).strip('-')
    return (s or 'task')[:max_len].rstrip('-')
```

This is intentionally NOT hash-suffixed: equal titles → equal slugs →
trivial dedup. Slight rewordings still create different slugs (which
is the v1 tradeoff). The classifier prompt explicitly tells AI to
reuse exact titles for the same conceptual task.

### 3.2 — Layer 1 (summarize.py) — move tasks to frontmatter

Schema gains a `tasks` field in the YAML frontmatter (matches the
artifacts/blockers pattern from DD-003):

```yaml
tasks:                              # see Rule 12. omit if none.
  - title: P15.3 HTML UI 加 modal + badge
    done: true
  - title: 推送 commit 到 origin
    done: false
```

Layer 1 emits `title` + `done`. The `id` is generated in Layer 2's
post-process so the slug function is single-sourced (classify.py).

The body section `# 任务（建议）` is **removed**. Tasks moving to
frontmatter is a clean break — the markdown form survived only
because nothing read it; once Layer 2 reads frontmatter, the body
form is dead weight that risks drift.

### 3.3 — Layer 2 (classify-cross-session.md) — §7c task aggregation

New rule, parallel to §7a / §7b:

```
§7c — Aggregate tasks for an initiative

INPUT: PRIOR initiative's `tasks[]` (each {id, title, done, ...}),
       hot summaries' `tasks[]` (each {title, done}).

ALGORITHM:
1. Union: every task from PRIOR + every task from any hot summary
   whose session belongs to this initiative.
2. Identity: a task is keyed by `id`. PRIOR tasks have ids; new tasks
   from hot summaries get an id generated by slugify(title) (described
   in §3.1 of DD-008).
3. Dedup: if multiple entries share an id, merge them:
   - `title`: most recent mention wins (latest by source's
     `last_activity_at`)
   - `done`: monotone — true if PRIOR was true OR any hot summary
     marked it true. Once true, never reverts.
   - `first_seen_at`: earliest known
   - `last_seen_at`: latest known
   - `done_at`: timestamp of the source that first marked it done
4. Sort: not-done first by `first_seen_at` desc, then done by
   `done_at` desc.
5. Cap visible to MAX_VISIBLE_TASKS=20:
   - All not-done tasks always kept (they're actionable).
   - Done tasks: keep most-recent up to remaining budget.
   - Overflow → archive (does not appear in `dashboard.json.tasks[]`
     but is preserved in `cache/task_archive/<init-id>.json`).
6. Output a `tasks_archived_count` field on each initiative (int).
```

### 3.4 — Deterministic post-process (`classify.py`)

After AI emits its mindmap, `classify.py` walks every initiative and
rebuilds `tasks[]` according to §7c — using the merged set of
{PRIOR tasks, hot-summary tasks, AI's emitted tasks}. AI output is
treated as a third evidence source, not authoritative. This is the
same pattern we already use for cold-initiative immutability and
artifact aggregation.

This means: even if AI ignores the new prompt rules, the post-process
catches it. Prompt + post-process is belt-and-suspenders.

### 3.5 — Archive file format

Path: `cache/task_archive/<initiative_id>.json`
(one file per initiative; safe filename derived from id)

```json
{
  "initiative_id": "claude-stray-dashboard",
  "updated_at": "2026-05-15T04:30:00Z",
  "tasks": [
    {
      "id": "p15-3-html-ui-modal-badge",
      "title": "P15.3 HTML UI 加 modal + badge",
      "done": true,
      "first_seen_at": "2026-05-14T22:00:00Z",
      "last_seen_at": "2026-05-15T03:42:00Z",
      "done_at": "2026-05-14T23:39:00Z",
      "sessions": ["940413c0-..."]
    },
    ...
  ]
}
```

Properties:
- **Full history**: every task ever seen for this initiative,
  including those evicted from the visible cap.
- **Idempotent**: regenerated from PRIOR archive + this round's
  observations on every classify run. Re-running is safe.
- **Mergeable across runs**: each task's `first_seen_at` is preserved
  forever; `last_seen_at` and `done_at` may advance.

Existing initiatives without an archive file: on first new classify
run, all current `dashboard.json.tasks[]` entries are treated as
"seen now", assigned generated ids, and seeded into the archive with
`first_seen_at = mindmap.generated_at` (best-effort fallback).

### 3.6 — UI surface

**Card head** (compact): the existing "tasks" badge gains an
archived-count suffix when non-zero:

```
✓ 3/5  +12 archived
```

Click opens the existing detail modal, which gains a new section:

**Detail modal — "Full task history"** (collapsible, default open
when tasks_archived_count > 0):

```
┌─ Tasks (3 active / 5 total visible / 17 archived) ─┐
│  · 推送 commit 到 origin                            │
│  ✓ P15.3 HTML UI 加 modal + badge                  │
│  ...                                                │
│                                                     │
│  ▾ View archived (17)                              │
│  ✓ P14.6 refresh-bg.sh 切到新 pipeline             │
│  ✓ P14.5 bin/classify.py + bin/layer2-trigger.sh  │
│  ✓ ... (timestamps shown)                         │
└─────────────────────────────────────────────────────┘
```

The archived list is fetched on demand from
`GET /api/task-history?init_id=<id>` so the main page payload stays
small. In static (non-server) mode, the archived list is silently
hidden.

### 3.7 — `serve.py` endpoint

New read-only endpoint:

```
GET /api/task-history?init_id=<id>
→ 200 {tasks: [{id, title, done, first_seen_at, last_seen_at,
                done_at, sessions}, ...]}
   404 if archive file doesn't exist
```

Reads `cache/task_archive/<id>.json` directly. No mutation.

## 5 — Migration

**Existing initiatives with N≤20 tasks**: nothing visible changes
(they fit within cap). On next classify run, archive file is created
with all current tasks seeded with `first_seen_at = generated_at`.

**Existing initiatives with N>20 tasks** (e.g. our claude-stray
case with 43 tasks): on next classify, post-process dedups +
caps. Visible drops to ≤20; the rest spill into the archive file.
Badge shows `+N archived`.

**Existing tasks without `id`**: post-process generates ids from titles
on first run. From then on, ids are stable.

**Schema bump**: `dashboard.json.schema_version` stays at 2 — adding
optional `tasks_archived_count` is backward-compatible. Tasks now
carry an optional `id` field; old readers ignore it harmlessly.

## 6 — Open questions

### 6.1 — Should `id` collisions be possible across initiatives?

Two initiatives can both have a task "推送 commit 到 origin". Slugs
will collide *within their own initiative's archive*, but across
initiatives ids are independent. This is fine — task ids are
initiative-scoped, like artifact urls used to be initiative-scoped
before DD-003 made them global.

### 6.2 — Fuzzy matching for near-duplicate titles

v1 dedupes by exact slug. "P14.2: 写 ... + 调优" and "P14.2: 写 ...
+ 3 个真实 session 调优" produce different slugs and stay as separate
tasks. v2 could add a normalized form (strip parenthesized clauses,
collapse "+ X" suffixes) for stronger consolidation. Deferred — v1
covers ≥80% of cases by anchoring AI to reuse exact titles.

### 6.3 — Archive growth over time

`task_archive/<id>.json` grows forever (additive). Even 5 years of
heavy use = thousands of tasks × ~200B each = ~1MB per initiative.
Not a concern for the local-only design. Out of scope: archive
compaction or tombstoning.

### 6.4 — Cap value (MAX_VISIBLE_TASKS = 20)

Hand-picked to match what fits in a card detail modal without
scrolling. Configurable via env: `CLAUDE_WORKTREE_MAX_VISIBLE_TASKS`.
