**补齐的空白**: 子卡的核心卖点「并行支线 → 合并回主干」此前完全没做(关闭流程只是
假设你已手动 `git merge`);也是对 Agent View(故意扁平、不记父子、无合并)的差异点。

## 为什么

黄金场景:主卡梳理出 N 件待办 → fan out N 张子卡并行推进 → 各自做完 → **一键合并回主干**。
此前 stray 只能并行、不能收口,合并全靠手动 git。本设计补上收口,且把「合并 + 解冲突」这种
脏活交给 AI agent(复用子卡基建),人类只在 agent 拿不准时一闸、在落地时一闸。

## 命题贴合

合并不是 stray 自己用 libgit 去做 —— 而是**再 spawn 一张子卡**,任务 = 「合并 X、解冲突」。
stray 只做编排(spawn + 串行 + 落地那一步 FF),execution(merge/解冲突)复用 Claude Code。
完全符合「triage + handoff,不重造执行」。

## 设计决策(Q1–Q7 收敛)

### Q1 合并方式 + 目标
- **本地 git merge**(不是 CR/MR;CR 是你之后用 a1 单独 ship 的下游步骤,不耦合进来)。
- **目标分支用户可选**,默认 = 主卡所在分支(`main_repo` 的 HEAD 分支,不一定是 `main`)。

### Q2→Q3 谁来合 + 怎么落地
- **合并 agent**:点合并 → spawn 一张**合并子卡**,自己的 worktree,分支 `merge-<slug>`(切自目标分支 HEAD)。
  它 `git merge worktree-<原slug>` + 解冲突 + commit → `merge-<slug>` = 目标 ⊕ 子卡成果、无冲突。
  解不动时**问用户**(卡转 needs_you)。
- **落地 = (C) 混合 FF**:目标分支正被主卡 checkout 占着,git 不许从外部更新被 checkout 的分支 ref。
  所以落地必须在主卡 checkout 里做一步 `git merge --ff-only merge-<slug>`(已无冲突 → 必然 FF、零风险):
  - 主卡工作树**干净** → stray 自动 FF;
  - 主卡有**未提交 WIP** → 不擅自动它,交还给你:「主卡先 commit/stash 再落地」。

### Q4 完成→落地的触发 + 合并 agent 生命周期
- **你点「落地」**(不轮询 git 判完成 —— agent 解冲突反复提交易误触发;且落地动主卡分支,本就该人类一闸)。
  合并 agent 当普通子卡呈现,你看它 done 了,点「落地到 <target>」,stray 才做那步 FF。
- 嵌在**主卡**下,标签「合并 ⊳ <子卡名>」。
- 落地成功 → **自动关掉合并 agent**(清它的临时 worktree + `merge-<slug>` 分支)。
- 解不动 → needs_you,你进它终端答一句,它继续。中途放弃 → 手动关。

### Q5 pre-check + 原子卡去向
- spawn 合并 agent 前三道闸门:① 原子卡分支相对目标**无新提交** → 拒「无可合并」;
  ② 原子卡有**未提交改动** → 警告「不会被合并,先去子卡 commit」,给「回去 commit / 就合已提交的」两条路;
  ③ 目标分支不存在 → 拒。
- **落地成功后自动关闭原子卡**(清 worktree+分支)—— 这就是用户授权的「子卡只能我手动删,或**合并功能去删**」。
  仍走现有关闭安全闸(原子卡会话还 live 就先拦)。

### Q6 UI 入口
- 「合并」按钮贴在每张子卡的「×关闭」旁,subrow + webterminal 会话族两处都加。
- 点「合并」→ 就地内联展开目标分支下拉(默认主卡分支)+「确认合并」(仿 DD-016 合并的内联样式)。
- 两阶段两按钮:普通子卡显示**[合并]**;合并 agent 干完后那行显示**[落地▸]**;需重合(本设计串行后基本不出现)。
  ```
  当前卡:DD-015 驾驶舱
    ▸ 改鉴权超时             [合并] [×]
    ▸ 写迁移文档             [合并] [×]
    ▸ 合并 ⊳ 改鉴权超时       [落地▸] [×]   ← 合并 agent,干完了
  ```

### Q7 并发 = 串行队列(关键简化)
- 点「合并」= **入队**;同一时刻**只处理一个**(串行锁)。
- 当前合并 agent:把队首子卡合到**当前**目标分支(已含前面刚落地的成果)→ 解冲突 → 落地 → 关原子卡 → 取下一个。
- 因为串行,**每次都从最新目标起步** → 不存在「base 过期、要重合」(并行方案才需要,已否决为过度设计)。
- 「一个没合完又点另一个」→ 自然**排队尾**。一个合并 = 一张合并 agent 子卡(符合「一卡一会话」)。

## 复用的基建
- 子卡 spawn / worktree / 嵌套 / 关闭 / live 状态 = DD-025 + DD-030(全复用)。
- `_worktree._git()` 跑所有 git 操作。
- 落地的 FF + 串行锁 + 队列 = 本设计新增的少量编排逻辑。

## 关键不变量
- **冲突永远在合并 agent 的 worktree 里解,绝不在主卡工作树里**。
- **落地永远只 FF,零冲突零风险**;主卡有 WIP 绝不擅自动它。
- **目标分支只前进、串行**;不丢任何已落地子卡的成果。

## 验证基线(交付即「可靠」)
- 单子卡:点合并→选目标→合并 agent 解冲突(必要时问你)→ 你点落地→主卡分支 FF 前进、原子卡自动关。
- 冲突:合并 agent 解;拿不准 → needs_you → 你答 → 它继续 → 落地。
- 多子卡:连点几个合并 → 串行一个一个落,每个从最新目标起,互不丢成果。
- 主卡有 WIP:落地被拦,提示先 commit/stash;commit 后落地成功。
