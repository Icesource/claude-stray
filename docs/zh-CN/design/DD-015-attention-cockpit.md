# DD-015: 注意力驾驶舱 —— 实时会话遥测 + 可交互控制

**Status**: Proposed —— **最高优先级**(作为主产品方向取代 DD-014 的 hero / 三层布局)
**Author**: Claude(与用户)
**Date**: 2026-06-01

英文原版：[../../design/DD-015-attention-cockpit.md](../../design/DD-015-attention-cockpit.md)

## 问题

我们发布过的每一种布局 —— 最初的卡片网格、以及 DD-014 的
hero / thread-card-chip 重设计 —— 都犯同一个产品错误:**它依然是
"一大堆"。** 用户原话:

> "无论是之前的卡片视图,还是现在优化后的,信息依然杂乱……通过 stray
> dashboard 我并不能感受到一个更简洁更清晰更完整上层的视图。之前的卡片
> 是一大堆,现在还是一大堆。"

> "问题的根源在于用户体验、信息分层、简洁、聚合、交互等等方面都不好,
> 它不一定是某个点去优化能解决的,我认为应该有一种更好的产品形态。"

根因**不是样式**。而是 stray 被当成"工作的*可视化*"来建("怎么把所有
initiative 都显示出来?"),这从结构上必然产出一面扁平、等权重、按分类法
排序的墙 —— 也就是把它本该消解的混乱,原样复现了一遍。产品**没有观点**,
于是把"扫描 + 排优先级"这件最累的事又甩回给了人,而这恰恰是它存在的意义
所要消除的痛。

另有两个结构性缺口,让它停留在"后视镜"而非"控制面":

1. **没有实时状态。** 我们只在 `Stop` hook 里事后总结。用户无法按会话
   看到:它现在在跑吗?跑完后空闲(等我)多久了?建议的下一步是什么?
   在长任务 / 高并发场景里最大的浪费 —— *一个 agent 干完活、闲在那等,
   而人不知道* —— 今天完全不可见。

2. **没法操作。** 面板只读。要对一张卡动手,用户得离开 stray、找到对应
   的终端 / 会话、手动重建上下文。多个并发工作之间切换成本很高,这直接
   压制了 stray 本应释放的并发度。

### 产品北极星(一切决策为之服务)

> 在 AI 干长任务的时代,给人一个**注意力驾驶舱**:一眼看清*哪些工作在
> 进行、每个进展到哪*,知道*此刻该管哪条*,并能*以近乎零成本切进任意
> 一条*去推进它。稀缺资源是人的注意力与决策 —— 产品的职责是分配它。

## 目标 / 非目标

**目标**

- 一个高空概览视图,**在不隐藏并发组合的前提下做到简洁** —— 所有*活着的*
  工作保持可见,只折叠*死掉的*(已完成 / 长期搁置)。
- 按**注意力状态**(等你 / AI在跑 / 闲置 / 完成)组织,而非按 workspace
  分类法。
- **按会话的实时**状态 + *闲置时长(idle-since)*计时,实时推送。
- 每会话一条**"下一步建议"**。
- **点卡片/会话 → web 终端**:展示最近对话,并能直接打字推进该会话,
  包括*接管一个正在运行的会话*(用户在 zellij 里跑 claude)。

**非目标**

- 不砍广度。明确拒绝"收件箱清零"框架(见"拒绝的方案")。简洁来自
  **密度 + 排序 + 视觉权重**,而非减少条目数。
- 本 DD 不重做身份/pipeline —— 纠正后的草图见下方"身份模型"小节;完整
  设计单独追踪(候选 **DD-016**)。
- 不是远程 / 多用户工具。严格 localhost、单用户。

## 原则(reframe)

1. **简洁 = 密度 + 排序 + 视觉权重,不是减少数量。** 旧墙之所以糟,是因为
   每张卡都是等大等重的胖块、在同一高度平铺、按 workspace 排序。12 条活跃
   工作*可以*一眼扫完 —— 只要每条压成一行高密度、按"该不该管我"排序、用
   视觉轻重区分(该管的吵、humming 的安静但在场)。广度和简洁不对立。

2. **项目/线为主空间分组,注意力为视觉权重 + 排序。**(原型后修正——最初
   "按注意力而非 workspace"过头了:扁平注意力档读起来像"所有会话堆在一起",
   而人是**按项目**切换上下文的。)按 workspace/cwd 分组——这是**稳定** key
   (目录,不是 AI 现编 id)。项目内按注意力排序 + 视觉加权(等你醒目、在跑
   安静、闲置变淡);项目之间按各自最紧急项排序,跨项目紧急事仍会浮现。扁平
   "按注意力"视图作为切换保留。见"原型产出"。

3. **Altitude / 渐进展开。** 默认 = 最高空(一行态势头条 + 高密度排序行);
   点开某行看完整上下文;再点进终端去操作。

4. **Artifact(MR / 需求 / PR / issue 链接)是可信的脊柱。** 它们是确定性
   抽取、人能看懂、且 —— 不像 AI 生成的名字/摘要 —— 不会在不同轮之间"飘"。
   所以:在每条工作行上显眼地渲染它们;用它们的存在作为*重要度/排序信号*
   (真实 MR + 需求 > AI 闲聊);并且 —— 一旦出现 —— 作为极强的*成员/归并
   提示*(某会话引用了已属于 initiative X 的 `MR!1234` → 归到 X)。它们是
   会话*过程中*才产生的,所以**不是身份 key**(见"身份模型")—— 它强化的是
   排序、归类和行脊柱,不是 id 创建。用户说这是唯一好用的功能 —— 顺着它做。

## 身份模型(review 后纠正)

本 DD **不**重做身份(完整设计 → 候选 DD-016),但之前那句简写"把 id 锚定到
session_id / artifact"在用户指出的两点上是错的,纠正后的模型记在这:

- **`session_id` 是不可变的原子**(会话出生即存在、永不变)。它是*底料*;
  initiative 的 id **不等于**它。
- **一个 initiative 是 session_id 的*持久化集合***,自带一个**代码生成、
  落盘一次**的 id(可由首个 session 派生)。→ *多 session 一任务是原生的* ——
  它就是这个成员集合本身。(回应反驳①)
- **AI 是归类器/打标器,不是 id 所有者。** 每轮它只回答(主要针对新 session):
  "从这个*固定 id 列表*里,它属于现有哪个 initiative,还是 new?" 已有成员
  关系冻结;合并/拆分是显式、有日志的操作。AI 永远不能给旧工作重铸新 slug ——
  而那正是今天卡片飘动 / 重复 / 复活的根因。
- **Artifact 不是身份 key。** 它们是会话*过程中*才出现的(反驳②),没法在
  出生时锚定身份。它们的角色是:(a) 排序/重要度信号;(b) 一旦出现就是极强的
  *成员/归并提示*。它强化归类和展示,不是 id 创建。
- **墓碑按成员 session_id 打,而不只是按 id**(归档路径已有时间窗口的 session
  墓碑;删除必须补上)—— 这样被删 initiative 里仍然"热"的 session 保持被
  tombstone,而不是被重新聚成一个新 slug。(顺带修掉诊断里那个"删了还活着"
  的 bug。)

**这只是收窄 AI 的*身份*权力,不是削弱它的智能。** AI 仍然拥有所有语义工作:
每会话总结、命名、进展、驾驶舱的状态判定(等你 / 在跑 / 阻塞)、`next_step`、
artifact 抽取,以及*提议*合并/拆分。我们只拿掉它"每轮铸造/重写 id"的能力 ——
那是它做得最差、又恰恰是今天不稳定根源的一件事。它的精力从"重新推导一套脆弱
的分类树(然后被后处理大半丢弃)"转向"无法被机械化的判断" —— 这是*更好地*用
AI,不是更少地用。

## 方案 —— 驾驶舱布局

> **这是信息线框,不是视觉稿。** 下面的 ASCII 只定*优先级 + 每行承载什么* ——
> 刻意不定外观。它读起来很糙是故意的,**不要照着字面实现**。排版、密度、
> 配色、动效在单独的高保真 HTML 原型环节里用真实 `dashboard.json` 数据敲定,
> 反复打磨后再锁视觉。

单一排序看板。所有*活跃*工作各占一行高密度,按注意力状态分档,档内按闲置
时长排序。死掉的工作折叠。

行结构:状态点 · **工作名** · 实时活动 / 卡点 / 闲置计时 · **artifact 链接
(脊柱)** · `[→]` 进终端。

**粒度。** 实时状态和终端天然是*按会话*的;而卡片是*会话的聚合*。处理:

- 卡片行显示**聚合**状态(取其会话中最紧急的:等你 > 在跑 > 闲置)。
- 展开卡片露出它名下的**活跃会话**,每个有自己的状态点和 `[→]`。
- 单会话卡的 `[→]` 直接进会话;多会话卡的 `[→]` 弹会话选择器(默认=最近
  活跃的那个)。

## 原型产出(2026-06-01)—— 已验证的决策

用真实 `dashboard.json` 数据做的高保真可交互原型在
`docs/design/prototypes/dd-015-cockpit/`(打开 `index.html`)。**交互模型已
定稿**,视觉打磨后续再说。已锁定:

- **两个视图(切换):** `按项目`(默认)、`按注意力`(扁平档)。
- **线级 rollup = 高空默认:** 每条项目/线收成一行摘要(件数 + 最急一句话 +
  各任务状态条 + 计时);最紧急的线自动展开。线 → 名下任务 → 任务详情+动作。
  (解决"还是一大堆"—— ~7 行,不是 23 个平级。)
- **分级:** `主线`(持续工作)vs `支线 · 个人/兴趣`;人/AI 可设。
- **可拖拽优先级:** 拖动排序、记忆;AI 可建议顺序(未接线)。
- **脑图导航**(`全部 → 项目 → 任务`):可折叠顶部总览、曲线连接、点节点跳转。
- **动效状态图标**(替代纯色点):转圈=在跑、脉冲圈=等你、空心=闲置、✓=完成、
  ⏸=搁置。
- **白天 + 夜间双主题**,可切换、记忆。
- 暂缓:进一步视觉/美学打磨。

## 能力 A —— 实时会话遥测(hooks → 状态 → SSE)

扩展 hook 集(今天只有 `Stop` + `SessionStart`)。每个 hook 往一个按会话的
实时状态文件 `cache/live/<session_id>.json` 追加;`serve.py` 监视该目录,
通过 **SSE** 推送变化给驾驶舱。

| 事件 | Hook | 写入 |
|---|---|---|
| 用户发了 prompt、AI 在干 | `UserPromptSubmit` | `state: running`,`started_at` |
| AI 一轮结束 | `Stop` | `state: idle`,`idle_since` |
| AI 要权限 / 要输入 | `Notification` | `state: needs_you`,`reason` |
| 会话开始 | `SessionStart` | 注册;捕获 **zellij 坐标**(`$ZELLIJ_SESSION_NAME`、`$ZELLIJ_PANE_ID`、cwd) |
| 会话结束 | `SessionEnd` | 注销 |

`idle_since` 驱动驾驶舱最重要的排序:在 idle / needs-you 档内,**等最久的
会话排最前** —— 这就是"别让干完的 agent 空等"的信号。

`serve.py` 新增 `GET /api/events`(SSE)。驾驶舱订阅一次,就地更新行(不再
全量重渲染 —— 顺带修掉前端诊断里那个"每次 poll 都 `board.innerHTML=''`
推倒重建"的问题)。每 8 秒轮询 `/api/data` 退役,改为推送。

风险面:**零新增 RCE 面。** 纯 hooks + 文件 + 只读 SSE。所以它最先发。

## 能力 B —— 每会话下一步建议

`bin/summarize.py`(Layer 1)本来就读整段会话。给它的 frontmatter 加一个
`next_step` 字段(改 `prompts/summarize-session.md`)。它显示在行上("建议:
跑集成测试 / 等你确认灰度策略")和终端头部。便宜、低风险。

## 能力 C —— web 终端(看最近对话 + 推进它)

点 `[→]` → 一个面板,**混合视图**:

- **上半**:从会话 jsonl 渲染出的最近 N 轮对话气泡(只读、好扫)。
- **下半**:实时终端(`xterm.js`)+ 输入框,接到真实会话。

两种接入模式,按实时状态:

- **会话正在跑**(用户在 **zellij** 里跑 claude —— 已确认)→ 用
  `SessionStart` 时捕获的坐标 attach 到它的 zellij pane。真正的双向镜像 +
  操控 live TUI。
- **会话空闲 / 已结束** → spawn `claude --resume <session_id>` 到新 PTY。
  resume 会回放上下文,用户打字继续。

**传输。** 推荐:spawn **`ttyd`**(极小的单文件二进制,`brew install ttyd`),
绑 `127.0.0.1` + 临时端口 + 一次性 token,前面接 `zellij attach …` 或
`claude --resume …`,嵌进面板。这样避免在 Python stdlib `http.server` 里
实现 WebSocket+PTY,也契合本项目"POSIX 工具、少重依赖"的取向。备选:自带一个
Python 的 WS↔PTY 桥(代码更多,少一个外部依赖)。建造时再定(见 Q3)。

## 安全模型(仅能力 C)

> **暂不考虑 —— 当前不是问题。** 阶段 1(遥测)和阶段 2(下一步)**零**安全
> 面,所以安全不阻塞优先路径。本节只在真正建造能力 C 时才适用,到时候再设计。

能力 C 把 serve.py 从**只读查看器**变成**能 spawn shell、能给 AI 发消息的
本地服务** —— 一个 localhost RCE 面。用户已明确接受这个权衡。必要管控:

- 严格绑 `127.0.0.1`;绝不 `0.0.0.0`。
- 每个终端一个**一次性 token** 放 URL 里;拒绝未认证的 WS / ttyd 连接。
- **默认关闭**,用显式 flag 开启(如 `stray --serve --enable-terminal`);
  flag 状态在 UI 上展示。
- 遥测(A)和下一步(B)留在只读安全路径上,不受此 flag 影响。

## 已锁定的决策(来自设计对话)

- **用户在 zellij 里跑 claude** → attach-to-live(能力 C 模式 1)可行,
  通过 zellij pane 坐标。
- **用户接受** stray 变成能 spawn PTY / 发消息的本地控制台(按上述方式
  gated)。
- 产品主轴是**注意力状态**,而非 workspace;保留广度(不做收件箱清零)。

## 迁移

- 实时状态文件是**增量的**(`cache/live/`);文件缺失 → 会话只是不显示
  实时状态(对升级前的会话优雅降级)。A/B 不需要改 dashboard.json schema。
- 驾驶舱读*同一份* dashboard.json initiatives;它是新的呈现,无需数据迁移。
  DD-014 的 `level`/`parent_thread_id` 字段变成分档的建议输入(或被丢弃 ——
  见 Q2)。
- 新 hook 幂等,可安全加到已有装置。

## 成本 / 风险

- **遥测/SSE**:token/CPU 成本可忽略;主要风险是 hook 漏触发导致状态过期 →
  加 heartbeat + "last seen",崩掉的会话衰减成 `unknown` 而不是谎称
  `running`。
- **next_step**:每条 Layer-1 摘要多几个输出 token。
- **终端**:真正的风险集中点(RCE 面、ttyd 依赖、zellij-attach 机制)。
  靠 flag 门控、localhost+token、最后才发 来缓解。
- **zellij attach** 机制(把 session_id 映射到 pane、在不打扰用户自己 zellij
  客户端的前提下 attach)需要 POC(Q1)。

## 拒绝的方案

- **收件箱 / 分诊(只显示"等你",其余折叠)。** 第一版提案。拒绝:收件箱
  以清零为理想,会隐藏并发组合,从而*降低*用户腾挪与选择的能力 —— 与目标
  相反。用户:"如果收件箱中呈现的任务太少,可能会降低并发度,用户也难以
  把控当前重要的事情。" 驾驶舱保留所有活跃工作,只折叠死掉的。

- **继续迭代 DD-014 的 hero/thread 布局。** 拒绝:DD-014 的 thread 档发布时
  **0 个成员被填充**(每个 thread 都是空壳),筛选/搜索只作用于 `.card`、
  根本筛不到 hero/threads/rail,精力花在视觉换皮上而真正的痛(没分诊、没
